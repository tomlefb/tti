# Look-ahead audit findings — TJR detector

**Date**: 2026-05-01
**Branch**: `feat/strategy-research`
**Audit script**: `calibration/audit_lookahead.py`
**Raw report**: `calibration/runs/lookahead_audit_<TS>.txt` (gitignored — regenerable from the script with `--seed 42 --n-samples 30 --n-dates 400 --start 2016-01-03 --end 2026-04-29`)

## Verdict

**SUSPECT** — the TJR detector uses future data when emitting historical
setups. **4 of 30** randomly sampled setups diverge between the
full-data run and a re-run on data truncated to `mss_confirm + 5min`.

This invalidates the unconditional use of historical backtests for
edge estimation: a fraction of the setups counted in those backtests
would never have been generated by the production scheduler at
real-time, or would have been graded differently.

Production go-live planned for Monday must be paused (or run with
explicit acknowledgment of this leakage) until the detector is fixed
and the backtests re-run.

---

## Method

For each sampled historical setup at MSS-confirm time `T`:

1. Slice all four OHLC frames so that `time <= T + 5min`. This
   represents the soonest moment the production scheduler — running
   on a 5-minute APScheduler tick — could have observed the new MSS
   candle.
2. Re-run `build_setup_candidates()` for the same target date with
   the truncated slices.
3. Look up the setup with the same
   `(symbol, killzone, direction, mss_confirm_candle_time, sweep_candle_time, swept_level_price)` key
   in the re-run output.
4. Compare every field used downstream (POI bounds, entry, SL, TP1,
   TP runner, RR, swept level, target level, quality, confluences).

Sample: 30 setups drawn from a pool of 57 (XAUUSD: 10, NDX100: 47),
seeded RNG, 400 trading dates per instrument over 2016-01-03 →
2026-04-29, 60-day rolling lookback per detection (matches the
extended-10y backtest). 26 setups reproduced bit-identically; 4 did
not.

## Root causes

### 1. **FVG forward-window is unbounded relative to detection time** (primary leak)

**Location**: `src/detection/setup.py:486-489`

```python
fvg_window_start = mss.displacement_candle_time_utc
fvg_window_end = mss.mss_confirm_candle_time_utc + pd.Timedelta(
    minutes=_FVG_LOOKFORWARD_FROM_MSS_MINUTES   # 30 min
)
```

The FVG search reaches **30 minutes past MSS confirm**. In real time
at `T + 5min`, only one M5 candle of that 30-min window has actually
formed; the historical run sees all six.

This causes the detector to find a tighter FVG that lives in the
`(T+5min, T+30min]` window, which a real-time scheduler would never
have seen. When that FVG is absent in the truncated re-run, the
detector falls back to `OrderBlock`, which has different
proximal/distal levels and therefore different entry, SL, and TP.

**Fields affected**: `poi_type`, `poi.proximal`, `poi.distal`,
`poi.size`, `poi.size_atr_ratio`, `entry_price`, `stop_loss`,
`tp1_price`, `tp1_rr`, `tp_runner_price`, `tp_runner_rr`,
`target_level_type`, `quality`, `confluences`.

**Concrete example (suspect 2)** — `NDX100 ny short 2016-01-12T16:40`:

| field            | full run                | truncated (T+5m)     |
|------------------|-------------------------|----------------------|
| poi              | `FVG`, prox=7245.5      | `OrderBlock`, prox=7237.5 |
| entry_price      | 7245.5                  | 7237.5               |
| tp_runner_rr     | 6.0                     | 3.55                 |
| quality          | B                       | B                    |

**Concrete example (suspect 4)** — `NDX100 ny short 2019-08-30T13:45`:

| field        | full run | truncated |
|--------------|----------|-----------|
| poi          | FVG      | OrderBlock|
| entry_price  | 10475.0  | 10492.5   |
| tp_runner_rr | 4.09     | 3.96      |
| **quality**  | **A**    | **B**     |

Suspect 4 is the worst case: the historical run grades the setup
**A** (notify-eligible per `NOTIFY_QUALITIES = ["A+", "A"]`); the
real-time run grades it **B** and would never have notified the
operator. Backtests that include this setup overstate the strategy's
A-grade edge.

---

### 2. **Sweep dedup pool spans the full killzone** (secondary leak)

**Location**: `src/detection/setup.py:337-346` — `detect_sweeps` is
called once per killzone with `killzone_window_utc = (kz_start, kz_end)`,
i.e. the full 3 h London or 2.5 h NY window. The dedup
(`src/detection/sweep.py:181+`) collapses near-duplicate sweeps to
the deepest one within a 30-min window.

**Leak**: With more total sweeps in the killzone, the "deepest"
representative inside a dedup cluster can change. As a result, an
early sweep that was preserved in the truncated run gets absorbed
into a later (post-T) sweep cluster in the full run, and vice
versa. The setup whose MSS confirms at `T` ends up tied to a
different upstream sweep — different swept level, different opposing
target, different RR.

**Fields affected**: `swept_level_price`, `swept_level_type`,
`swept_level_strength`, `sweep.return_candle_time_utc`,
`target_level_type`, downstream entry/SL/TP/RR.

**Concrete example (suspect 1)** — `NDX100 london long 2016-05-18T07:25`:
the historical run sweeps `7257.25` (asian_low, structural); the
truncated run sweeps `7258.00` (swing_h1_low, minor) at sweep candle
`07:15`. Same MSS confirm candle, different originating sweep — the
historical setup’s sweep doesn't appear in the truncated re-run at
all (1 setup total in re-run, with a different sweep identity).

---

### 3. **Setup completely missing in the truncated re-run** (composite leak)

When the FVG-forward-window leak (cause 1) and/or sweep-dedup leak
(cause 2) interact with `MIN_RR` and the grade ladder, the truncated
re-run can produce **zero setups** at the historical MSS confirm
time. The historical backtest counted this trade; the production
system never would.

**Concrete example (suspect 3)** — `NDX100 ny long 2024-06-14T13:50`,
graded **A** in the historical run, **0 setups** in the truncated
re-run.

This is the worst category: an A-grade setup, which under
`NOTIFY_QUALITIES = ["A+", "A"]` enters live execution since
Sprint 7, was never producible by the production code path. Every
such setup in a backtest is pure phantom edge.

## Quantified impact estimate

From the audit sample (30 setups, seed=42):

| Bucket                              | Count | %     |
|-------------------------------------|-------|-------|
| Bit-identical                       | 26    | 86.7% |
| FVG → OrderBlock fallback           | 2     |  6.7% |
| Different sweep selected            | 1     |  3.3% |
| Setup absent in real-time re-run    | 1     |  3.3% |

Of the 4 suspects, 1 changed quality A → B (would never have
notified) and 1 went from "exists" to "does not exist". The other 2
were B-grade in both runs (notification gate not affected, but TP/RR
inflated in the historical run).

Sample size is small but the failure mode is structural: the FVG
window is hard-coded to `+30min`, so any setup whose chosen FVG
crystallises in `(T+5m, T+30m]` will leak. A larger sample would
likely find broadly similar rates.

**Expected backtest distortion** (signed estimates, not measurements):

- **Setup count**: overstated by a few percent (phantom setups like
  suspect 3 that never produce in real time).
- **A-grade share / notify rate**: overstated (some real-time B's
  promoted to A by post-hoc FVG selection).
- **Mean R**: overstated. Tighter FVGs in the historical run yield
  lower SL distance → higher RR → higher win-conditional R. The 91.5
  point TP-runner gap in suspect 4 (NDX100, 13.6× the typical sweep
  buffer) is illustrative.
- **Drawdown**: understated by the same mechanism.

The Sprint 6.6 backtest validation that produced the live
`WATCHED_PAIRS = ["XAUUSD", "NDX100"]` portfolio is therefore not
clean — it may have admitted instruments whose real-time edge is
weaker than the backtest implied. The conclusion (XAUUSD + NDX100 are
A/A+ tradable) is not necessarily wrong, but it has not been verified
on a leak-free detector.

## Fix sketch (not implemented)

The detector needs an explicit "as-of" time analogous to
`as_of_utc=kz_start_utc` in `mark_swing_levels`, but propagated to
every component that touches data after MSS confirm:

1. **FVG window cap**: bound `fvg_window_end` by
   `min(mss_confirm + 30min, now_utc)`. In production, `now_utc` =
   the scheduler tick time. In backtests, `now_utc` = `mss_confirm + 5min`
   (the minimal real-time latency under a 5-minute APScheduler).

2. **Sweep dedup pool**: scope the sweep detection / dedup to
   `killzone_window_utc = (kz_start, min(kz_end, now_utc))` so the
   dedup pool is identical to what was visible at `now_utc`.

3. **Backtest harness**: instead of one detector call per killzone
   (current), iterate the scheduler tick by tick — for each 5-min
   tick within the killzone, call `build_setup_candidates(..., now_utc=tick)`
   and accumulate any new setups. This makes the backtest a true
   simulation of the live scheduler (the requirement stated in
   CLAUDE.md rule 11: lifecycle parity).

4. **Setup identity at completion**: once a setup is committed at a
   given tick, lock its identity and do not re-emit it on later ticks
   even if a tighter FVG appears. (Mirrors live behaviour where the
   setup is sent to the operator the moment MSS confirms.)

A simpler short-term mitigation, if the full tick-by-tick simulation
is too costly, is just step 1 — the FVG cap. That alone should
remove the dominant cause of the 2/30 FVG→OB divergences and
suspect 4’s A→B grade flip.
